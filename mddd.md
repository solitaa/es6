Introduction
------------

### What is the Exchange API?

The Rubicon Project Exchange API provides a standard solution with which any ad server or ad-rendering SDK can integrate directly with the Rubicon Project’s real-time exchange. The Rubicon Project is currently integrated with more than 100 DSPs servicing over 100,000 advertisers. The Exchange API exposes these buyers through Rubicon Project’s auction for use in your ad server. Via this integration, your ad server gets direct access to a complete, mature and competitive RTB marketplace.

Fully integrating a mature exchange into your ad server enables optimal yield by maximizing the competition from indirect and direct demand for every impression. The API also provides support for private marketplaces (via Deal ID) and emerging Programmatic Guaranteed (PG) deals.

Integrating the Exchange API eliminates the need for a tag-based solution reducing discrepancies, passbacks and latency associated with this conventional approach.

### Common Use Scenarios

While the Exchange API is built to be quite flexible and adaptable based on the approach of the ad server or ad rendering SDK, some common use scenarios are as follows:

*   Open Marketplace – Soliciting Rubicon Project’s open marketplace as a source of additional demand that can compete with line items in the ad server in real time on a CPM basis.
*   Private Marketplace – Deals can be formed with buyers integrated with Rubicon Project, with special terms agreed between sellers and buyers, and executed programmatically. It can be beneficial to ensure these deals are prioritized effectively within the ad server.

Furthermore, these scenarios are supported across the major ad formats and ad delivery environments:

*   Mobile (web and app)
*   Video
*   Display

Objective
---------

This document describes the API supported by the Rubicon Project’s real-time exchange for server-side calls to request ads. The API is an HTTP POST call that supports both JSON or Protobuf request bodies, responding in turn with a JSON or Protobuf document, respectively. This API is explicitly designed to look as much as possible like OpenRTB while implementing only a subset of OpenRTB. All differences from OpenRTB are detailed in the specification.

The following terms are used throughout this document specifically in the context of the Exchange Interface and this specification.

### HTTP Headers

#### Authentication

Calls to the Exchange API require basic HTTP authentication. HTTPS can be used in conjunction with basic HTTP authentication. See [this link](http://en.wikipedia.org/wiki/Basic_access_authentication) for detail on how to set this header. The username and password is established between the partner and the Rubicon Project when the partner is ready to start integration testing.

Please note that it is recommended that the caller set the Authorization header proactively, rather than waiting for an HTTP 401 response. This will avoid unnecessary load at both ends of the connection.

#### User Agent

It is important that the caller set the user agent header on the request: the Rubicon Project uses the user agent string to identify and block requests from suspected web spiders, and not setting the user agent string explicitly, or at all, may trigger this filter. Suggested Implementation Logic
------------------------------

This API applies to ad servers that have access to make source-code level changes to the core decisioning engine of the ad server. This API can enable the ad server to integrate the Rubicon Project Exchange natively into the decisioning engine. The API is best served as the exclusive API for integrating an RTB exchange.

1.  Primary ad server receives the impression from the client.
2.  Impression is parsed and interpreted as normal.
3.  Targeting and ad decisioning occurs with local demand.
4.  A tentative winner and runner-up are chosen.
5.  The Exchange API is called with the following pricing elements: – inventory floor price (bidfloor) – tentative winner price and ID (see [Proxy Demand Object](#proxy-demand-object)) – tentative runner-up price and ID (see [Proxy Demand Object](#proxy-demand-object)).
6.  The Rubicon Exchange receives the impression and interprets it.
7.  All publisher configuration is retrieved and applied.
8.  The Exchange solicits bidders for bids and validates them.
9.  The Exchange runs the Unified Auction.
10.  The response is formatted and sent back to the Primary ad server.
11.  The ad server processes the response and makes a final decision.
12.  The winner ad is served to the client.

### Outcome Cases and Yield

Case 1: Proxy demand wins the Unified Auction. The ad server will receive this in the response with the ID of the winner.

Case 2: No bids exceed the bidfloor, no proxy demand passed in. The ad server may return HTTP 204.

Case 3: Exchange Open Market demand matches or beats the bid floor and wins the Unified Auction. The ad server will receive this in the response and should consider this winner in the final decision as ‘non-guaranteed’ demand and make an appropriate yield based decision balancing pacing of the tentative winner, inventory scarcity, and yield from the exchange winner.

Case 4: Exchange Private Market demand matches or beats the bid floor and wins the Unified Auction. The ad server will receive this in the response, marked with a DealID, and should consider this winner in the final decision as low-priority guaranteed demand and make an appropriate yield based decision balancing pacing of the tentative winner, inventory scarcity, yield from the exchange winner, and the goals of the DealID.

User Sync Details
-----------------

### 3.1 Multi-sync

A key part of Rubicon Project’s ad serving platform is managing state of ad delivery (i.e. freq-caps) and per-user data (i.e. user identifiers). Both rely upon browser cookies which will not be available when called from an Exchange API partner’s servers. To resolve this issue, Rubicon Project has developer a client-side iframe solution called Multi-sync.

Multi-Sync initiates the process of sending or receiving (or both) of user identifiers, called User Sync, between Rubicon Project and various types of partners such as DSPs, DMP and server side sellers / partners.

Exchange API partners must supply a URL that the Rubicon system will redirect the Rubicon Project user ID to. The handler should parse the value and store it subsequent XAPI calls (see Sections 3.1.2 and 3.2).

### Multi-Sync Endpoint

Protocol

URL

secure

https://eus.rubiconproject.com/usync.html

### Multi-Sync Query Parameters

Query Parameter

Description

p

Partner key assigned by Rubicon Project, identifying Partner.

endpoint

Exchange API calling data center for user. Value must be one of: “us-east”, “us-west”, “eu” or “apac”.

### 3.1.1 Recommended implementation

Rubicon Projects recommends multi-sync be initiated in the DOM at the end of page load, after all content has been delivered, independent of ad serving, once per page. Multi-sync has a built in frequency cap to sync with XAPI partners at a current rate of once per day, while syncing with a limited number of DSPs per instance, up until all DSPs have fully synchronized. After which, no further syncs with DSPs will occur until the ttl expires (set by the DSP) or the cookie is deleted.

Protobuf details

[Protocol Buffers](https://en.wikipedia.org/wiki/Protocol_Buffers) (aka protobuf) is an open-source way to serialize and deserialize structured data. It uses a binary message format (vs. human readable text like JSON), making it more computationally efficient and much smaller in size (up to a 50% reduction). A basic introduction to Protobuf is here: [Google Protocol Buffers.](https://developers.google.com/protocol-buffers)

More information on how to get started with protobuf, implementation details, and additional resources including IAB and Magnite XAPI specific definitions can be found [here](https://resources.rubiconproject.com/resource/magnite-internal/exchange-api-xapi/xapi-protobuf-integration/).

User Privacy Flags
------------------

There are three major flags that can be sent into the API to control aspects of user privacy. Each of them is described below.

### 5.1 Limit Ad Tracking

“Limit Ad Tracking” is a setting available in mobile and connected devices that allows users to opt-out of interest-based ads. It is called “Limit Ad Tracking” on iOS and Roku, “Opt-out of Interest Based Ads” on Android, and “Interest-based Ads” on Amazon Fire TV.

The Exchange API supports this flag and will pass it on to bidders and DSPs.

### 5.2 Do Not Track

Do Not Track is a technology embedded in the user’s web browser that enables users to opt out of tracking by websites they do not directly visit, including analytics services, advertising technology vendors, and social platforms.

See [donottrack.us](http://donottrack.us/) for more details.

The Exchange API supports this flag and will pass it on to bidders and DSPs.

### 5.3 COPPA

The USA’s Children’s Online Privacy Protection Act (COPPA) is a rule change implemented by the Federal Trade Commission in July 2013 that restricts almost all forms of tracking across child-directed sites and apps, other than for a set of limited “internal operations purposes.” This is interpreted to apply to analytics services, advertising technology vendors, and social platforms. Child-directed websites and apps may declare themselves as being under COPPA regulations since OpenRTB 2.2.

The Exchange API supports this flag and will pass it on to bidders and DSPs.

### 5.4 22580

In late 2013 USA’s California legislature passed a law concerned with the use of the internet by minors in that State. Part of this legislation prohibits certain products from being advertised to these audiences.

See California’s Legislative Information page on [Cal Bus & Prof Code §22580](http://leginfo.legislature.ca.gov/faces/billNavClient.xhtml?bill_id=201320140SB568)
